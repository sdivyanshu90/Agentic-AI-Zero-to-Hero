### 4.3 — Tool Execution Sandbox

> Concept: Generated code runs inside a disposable Docker container with no network, strict resource limits, and no host mounts.
> Attack surface: Arbitrary Python generated by the agent attempts filesystem reads, network egress, or container escape.
> Eval target: Isolation success rate > 0.99 on 30 exploit payloads

---

## Section 1 — Core Intuition

If the agent can execute arbitrary code on the host, the host becomes part of the prompt and every secret, socket, and mount is now in scope for exploitation. The mechanism here is containment, not trust: generate code, run it in a short-lived sealed container, and return only stdout, stderr, and exit code; the analogy is a biological sample in a sealed test chamber, because without the chamber the failure mode is not a bad answer, it is host compromise.

ATTACK: Host filesystem read
VECTOR: Generated code tries to read sensitive files using Python file APIs or shell commands.
EXPLOIT PAYLOAD: `open('/etc/passwd').read()`
CONSEQUENCE: Secrets leak from the host into the agent output.
MITIGATION:

Generated code runs in a Docker container with explicit resource limits and automatic removal after execution:
```python
container = docker_client.containers.run(
    image="python:3.12-alpine",
    command=["python3", "-c", code_text],
    mem_limit="128m",
    cpu_period=100000,
    cpu_quota=50000,
    network_mode="none",      # no network access
    read_only=True,           # no filesystem writes
    remove=True,              # deleted after execution
    timeout=10,
)
```

## Section 6 — Pitfalls & Pro-Tips

• FAILURE: Teams mounted the repo into the sandbox for convenience, which turned a code runner into a host file browser.
FIX: `output = docker_client.containers.run(..., volumes={}, read_only=True)`
• FAILURE: Repair loops retried forever after deterministic syntax errors and quietly exhausted the budget.
FIX: `if state["retry_count"] >= 3: return "fail"`
• FAILURE: Network egress remained enabled because engineers assumed the container had no useful route out.
FIX: `output = docker_client.containers.run(..., network_mode="none", ...)`
• FAILURE: A writable root filesystem let payloads stage large temporary files and trigger disk pressure.
FIX: `output = docker_client.containers.run(..., read_only=True, mem_limit="64m", ...)`
• FAILURE: Sandbox failures bubbled up as raw exceptions, which killed the whole agent instead of the execution branch.
FIX: `except ContainerError as exc: return ToolError(tool_name="execute_code_in_sandbox", error_type="ContainerError", message=str(exc))`
