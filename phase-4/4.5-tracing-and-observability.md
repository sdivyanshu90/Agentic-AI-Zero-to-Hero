### 4.5 — Tracing & Observability

> Concept: Full-fidelity conversation traces capture every LLM call, tool call, retry, error, and token count for post-incident diagnosis.
> Attack surface: Missing or leaky traces hide production failures or turn observability data itself into a secrets exposure channel.
> Eval target: LLM call capture rate = 1.0 on 50-call stress test

---

## Section 1 — Core Intuition

An agent that cannot be traced is indistinguishable from a rumor during an incident, because you do not know which model ran, which tool failed, or where the latency budget went. The mechanism is explicit span instrumentation around every meaningful boundary and asynchronous export to a trace backend; the analogy is a flight data recorder, because without it the postmortem becomes guesswork and the same outage repeats.

ATTACK: Secret leakage in traces
VECTOR: Prompt, tool args, or environment-derived values are logged verbatim into spans.
EXPLOIT PAYLOAD: `Authorization: Bearer sk-live-secret` stored as a span attribute.
CONSEQUENCE: The observability backend becomes a second copy of your secrets.
MITIGATION:

Every LLM call is wrapped in an OpenTelemetry span. Spans carry model name, token counts, and latency as attributes so operators can query them in any OTLP-compatible backend:
```python
with tracer.start_as_current_span("llm.completion") as span:
    span.set_attribute("llm.model", model_name)
    span.set_attribute("llm.prompt_tokens", prompt_tokens)
    response = client.chat.completions.create(...)
    span.set_attribute("llm.completion_tokens", response.usage.completion_tokens)
    span.set_attribute("llm.latency_ms", int((time.time() - t0) * 1000))
```


ATTACK: Sync exporter outage amplification
VECTOR: Trace export happens on the request path, so collector slowness directly increases user latency.
EXPLOIT PAYLOAD: OTLP collector stalls for 2 seconds per batch.
CONSEQUENCE: The observability plane becomes the outage source.
MITIGATION:

See the implementation in [4.5-tracing-and-observability/from_scratch.py](4.5-tracing-and-observability/from_scratch.py) and [4.5-tracing-and-observability/production.py](4.5-tracing-and-observability/production.py) for the concrete mitigation logic applied at the boundary described above.


ATTACK: Span omission hides tool abuse
VECTOR: Engineers instrument only the LLM call and never emit child spans for tool retries.
EXPLOIT PAYLOAD: Repeated failing tool calls inflate latency while the trace still looks healthy.

## Section 6 — Pitfalls & Pro-Tips

• FAILURE: Teams used `SimpleSpanProcessor`, and every trace export sat on the request path during load spikes.
FIX: `provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter(endpoint=os.getenv("OTEL_EXPORTER_OTLP_ENDPOINT"))))`
• FAILURE: Tool retries were invisible because only the top-level conversation span existed.
FIX: `with tracer.start_as_current_span("tool_call") as span:`
• FAILURE: Postmortems could not separate model latency from tool latency because both were dumped into one undifferentiated span.
FIX: `with tracer.start_as_current_span("llm_call") as span:`
• FAILURE: Observability data itself leaked secrets because prompts were shipped to the backend unredacted.
FIX: `sanitized_prompt = prompt.replace(os.getenv("OPENAI_API_KEY"), "[REDACTED_API_KEY]")`
• FAILURE: High-token conversations never triggered alerts, so cost regressions arrived in the billing dashboard days later.
FIX: `state["needs_alert"] = state["latency_ms"] > 5000 or state["prompt_tokens"] > 20000`
