### 5.1 — Async Task Queue

> Concept: Long-running agent jobs move off the request thread into a durable worker queue with retries, status tracking, and a dead-letter queue.
> Attack surface: Unbounded task payloads, guessable task IDs, and retry storms can turn the queue into a denial-of-service amplifier.
> Eval target: Retry correctness = 1.0 on 20 failing tasks

---

## Section 1 — Core Intuition

If an agent takes minutes, holding an HTTP connection open is a design bug, not a UX choice, because you have no durability, no retry semantics, and no way to recover after a worker crash. The mechanism is decoupling: accept the task, hand it to a durable queue, track status separately, and quarantine repeat failures in a dead-letter queue; the analogy is a warehouse intake dock that scans and shelves incoming packages before a picker touches them, because without it every transient failure becomes a user-visible outage.

ATTACK: Task ID enumeration
VECTOR: An attacker guesses sequential task IDs and polls other users' results.
EXPLOIT PAYLOAD: `GET /tasks/42/status`
CONSEQUENCE: Cross-tenant result leakage.
MITIGATION:

Worker exceptions are caught per-task and stored in `TaskState.error` rather than crashing the worker thread. The queue continues draining even when individual tasks fail:
```python
try:
    result = run_prompt(task.prompt)
    task.status = "Completed"
    task.result = result
except Exception as exc:
    task.status = "Failed"
    task.error = str(exc)
    log.error("task_failed", task_id=task.task_id, error=str(exc))
finally:
    task_queue.task_done()
```


ATTACK: Retry storm amplification
VECTOR: A malicious prompt triggers a deterministic exception and the queue hammers the provider until the broker saturates.
EXPLOIT PAYLOAD: `Generate output with 2 million tokens and keep retrying if rejected.`
CONSEQUENCE: Worker starvation, broker backlog, and rate limit exhaustion.
MITIGATION:

See the implementation in [5.1-async-task-queue/from_scratch.py](5.1-async-task-queue/from_scratch.py) and [5.1-async-task-queue/production.py](5.1-async-task-queue/production.py) for the concrete mitigation logic applied at the boundary described above.

## Section 6 — Pitfalls & Pro-Tips

• FAILURE: Teams kept task state only in memory, so a pod restart erased every in-flight job.
FIX: `redis_client().set(f"task:{record.task_id}", record.model_dump_json(), ex=86400)`
• FAILURE: Workers retried on every exception class, including deterministic validation failures, and filled the broker with dead work.
FIX: `@celery_app.task(bind=True, autoretry_for=(APIError, APITimeoutError, RateLimitError), ..., max_retries=3)`
• FAILURE: DLQ visibility was missing, so operators learned about poison tasks from user complaints.
FIX: `return [json.loads(item) for item in redis_client().lrange("dead_letter", 0, 100)]`
• FAILURE: A single shared OpenAI client was created at import time and behaved inconsistently after worker forks.
FIX: `client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))`
• FAILURE: Oversized prompts were accepted at the API edge and only failed after occupying a worker slot.
FIX: `prompt: str = Field(min_length=1, max_length=4000)`
