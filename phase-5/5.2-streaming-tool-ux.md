### 5.2 — Streaming Tool UX

> Concept: Intermediate tool and model events stream to the frontend over Server-Sent Events instead of waiting for the final answer.
> Attack surface: Unescaped event payloads, proxy buffering, or dropped stream ordering can turn streaming into a misleading or exploitable channel.
> Eval target: Time-to-first-token < 500ms on 30 streaming runs

---

## Section 1 — Core Intuition

Users do not need fake reasoning transcripts, but they do need proof that the system is progressing, which means surfacing real intermediate events as they happen rather than hiding the workflow behind a spinner. The mechanism is straightforward: stream typed events for tool start, tool end, agent text, error, and completion; the analogy is package tracking with real scan events instead of a single delivery confirmation, because without it every slow request looks identical to a dead one.

## Section 5 — Evaluation & Security

The harness scores event ordering and terminal delivery, because the user experience fails if a `done` event arrives before the last tool result even when the backend thinks the request succeeded. The adversarial cases attack the transport itself: newline injection, empty prompts, and oversized outputs.

ATTACK: SSE injection
VECTOR: Event data includes raw newlines, which lets a malicious payload smuggle a second event into the browser stream.
EXPLOIT PAYLOAD: `agent_message\n\nevent:error\ndata:pwned`
CONSEQUENCE: The client renders forged events or terminates the stream early.
MITIGATION:

Stream events are typed and carry a monotonic `sequence_id`. Clients can detect gaps and duplicates, and the `done` event signals clean termination:
```python
class StreamEvent(BaseModel):
    event: Literal["tool_start","tool_end","agent_message","error","done"]
    data: str
    sequence_id: int = Field(ge=0)
# Generator increments sequence_id for each event:
async def event_stream(prompt: str) -> AsyncIterator[StreamEvent]:
    seq = 0
    yield StreamEvent(event="tool_start", data="...", sequence_id=seq); seq += 1
    ...
    yield StreamEvent(event="done", data="", sequence_id=seq)
```


ATTACK: Proxy buffering
VECTOR: nginx or ALB buffers the SSE stream and flushes only after completion.
EXPLOIT PAYLOAD: Default proxy config with buffering enabled.
CONSEQUENCE: Users see no intermediate events and assume the agent is dead.
MITIGATION:

See the implementation in [5.2-streaming-tool-ux/from_scratch.py](5.2-streaming-tool-ux/from_scratch.py) and [5.2-streaming-tool-ux/production.py](5.2-streaming-tool-ux/production.py) for the concrete mitigation logic applied at the boundary described above.

## Section 6 — Pitfalls & Pro-Tips

• FAILURE: Teams streamed plain text chunks and the frontend guessed which ones were tool events, producing inconsistent rendering across models.
FIX: `class StreamEvent(BaseModel): event: Literal["tool_start", "tool_end", "agent_message", "error", "done"]`
• FAILURE: Proxy buffering made the stream look broken even though the backend was emitting events correctly.
FIX: `headers={"X-Accel-Buffering": "no", "Cache-Control": "no-cache"}`
• FAILURE: Newlines inside event data created forged SSE frames in the browser.
FIX: `encoded_data = json.dumps(event, ensure_ascii=True).replace("\n", "\\n")`
• FAILURE: Time-to-first-token regressed after a logging change because no one recorded it as a first-class metric.
FIX: `yield StreamEvent(event="tool_end", data=f"ttft_ms={int((time.perf_counter() - started_at) * 1000)}", sequence_id=sequence_id)`
• FAILURE: Streams ended without a terminal event on provider disconnects, leaving the client hanging forever.
FIX: `yield {"event": "done", "data": "stream_complete", "id": str(event_index)}`
